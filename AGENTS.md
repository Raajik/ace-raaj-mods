### 8.1 Conventions / Warnings
- **`ACE.Common.ThreadSafeRandom.Next(int min, int max)` is inclusive on both bounds** (internally calls `Random.Next(min, max + 1)`). Using `array.Length` as the max argument causes `IndexOutOfRangeException`; always use `Length - 1` for zero-indexed array access.
- **Prefer `nameof(Target.Method)` for Harmony patch targets.** String-based targets (e.g., `"get_MerchandiseBuyRate"`) silently fail at runtime if the method signature changes; `nameof` fails at compile time and is refactor-safe.
- **Cross-mod integration: verify method visibility before assuming callable.** The unified event scheduler (`EventScheduler.cs`) required changing `InvasionRuntime.StopWaveInternal` and `CullRuntime.TryStartCull` from `private` to `internal`. Always check access modifiers when wiring new callers across files.
- **Generic collection variance: `List<int>` and `List<uint>` are incompatible with `??`.** The null-coalescing operator requires matching types; explicit `.Select(lb => (uint)lb).ToList()` is required when mapping landblock IDs between `int` and `uint` representations.
- **Dual-path lifecycle: preserve legacy behavior behind a toggle.** The `UseUnifiedScheduler` setting allows both independent timers (legacy) and centralized scheduling (new) to coexist. When refactoring event systems, gate new behavior so existing deployments aren't broken.
- **Deprecate settings gracefully — don't remove properties from `Settings.cs`.** Existing `Settings.json` files will break if properties are removed. Instead, keep the property, update the doc comment to `DEPRECATED`, and stop using it in code. Example: `ExtraRollsTrained` / `ExtraRollsSpecialized` in `TrophyDropsSettings`.
- **Use `IPlayer.MonarchId` for lightweight allegiance checks.** No need to call `AllegianceManager.LoadPlayer()` if you only need to test same-allegiance — compare `player.MonarchId == patron.MonarchId`.
- **`Character.HasAsFriend(friendId, lock)` extension exists for friend checks.** Use `player.Character.HasAsFriend(patron.Guid.Full, player.CharacterDatabaseLock)` instead of manually iterating `CharacterPropertiesFriendList`.

- **Folder organization preference** — Use subfolders to keep content tidy (e.g., `SpellSiphon/Content/SQL/`, `WorldEvents/Content/`). Default to nested directories over flat structures when adding new asset types.
- **Trained/Specialized Equivalence** — When a user says a skill needs to be "trained" for a feature, being **specialized** also satisfies the requirement (specialized is a superset of trained). Code checks should use `>= SkillAdvancementClass.Trained` rather than `== Trained`.
- **`[ThreadStatic]` for prefix→postfix state communication** — When a prefix needs to signal a postfix (e.g., "this was a chaos failure"), use `[ThreadStatic] static bool _wasAction`. Reset it at the start of the prefix. This avoids method signature changes and works across Harmony patch boundaries. Example: `ChaosTinker.cs` uses `_wasChaosFailure` to track forced failures from prefix to postfix.
- **Patching `CreatureAttribute.GetCurrent(bool)`, not `.Current` property** — The `Current` property getter on `CreatureAttribute` is a compiler-generated wrapper; Harmony postfixes on it may not fire reliably. Patch `GetCurrent(bool enchanted)` instead with explicit `Type[] { typeof(bool) }` parameter types. Same applies to `CreatureSkill.Current` — prefer patching the getter method body directly or use `GetCurrent` where available.
- **Accessing private `creature` field in ACE entity classes** — `CreatureSkill`, `CreatureAttribute`, and `CreatureVital` all store a private `creature` field. In Harmony postfixes, use `Traverse.Create(__instance).Field<Creature>("creature").Value` to access it. This is required when patching these classes to apply player-specific logic.
- **`target.GetProperty(PropertyInt.NumTimesTinkered) ?? 0` vs `target.NumTimesTinkered ?? 0`** — `NumTimesTinkered` is a `PropertyInt` accessed via `GetProperty()`; it is NOT a nullable auto-property. Using `??` on `int` causes `CS0019`. Always use `target.GetProperty(PropertyInt.NumTimesTinkered) ?? 0`.
- **Verbose result tracking pattern** — When a method spends/allocates/consumes resources across multiple targets, return a `record` or `readonly record struct` with per-target details instead of a scalar count. This enables rich messaging without duplicating the allocation logic. Example: `SpendResult(string Name, int RanksGained, long XpSpent)`.
- **`WorldObject` is abstract — use `WorldObjectFactory.CreateNewWorldObject(weenie)`** — Direct instantiation with `new WorldObject(weenie, guid)` causes `CS0144` (cannot instantiate abstract class). Always use the factory. For realm builds, the factory overload may need a ruleset parameter.
- **`Vendor.DefaultItemsForSale` is a readonly field** — Do not use `??=` or direct assignment. It is initialized by ACE vendor setup. Only call `.Add()`, `.Remove()`, `.Clear()`, or read from it.
- **Account-wide JSON persistence pattern** — For cross-character state (e.g., quest completion tracking), use a static class with `Dictionary<uint, ...>`, `Load()` on startup, `Save()` on change, and a file path derived from `Assembly.Location`. This matches `AccountQuestFlags.cs` and `AccountRepeatQuestTracker.cs`.
- **`TimeSpan.GetFriendlyString()` for ACE-style duration formatting** — ACE.Common.Extensions provides `GetFriendlyString(this TimeSpan)` which produces strings like "2h 15m 30s". Use this for player-facing cooldown messages instead of manual formatting.
- **repeatQB system pattern** — For quest repeat rewards that should count as "new solves" for QP calculation, create unique ACE quest registry entries with a suffix (e.g. `{baseName}#repeatQB{n}`) rather than maintaining a separate stamp multiplier pool. This naturally integrates with existing QP counting logic. Use `GetBaseQuestName()` to strip the suffix for dictionary lookups.
- **Kill task auto-reaccept pattern** — Patch `QuestManager.HandleKillTask` prefix to detect max-solved + cooldown-expired state, call `SetQuestCompletions(name, 0)` to reset, and store timestamp in a JSON `PropertyString`. Postfix detects fresh max-solve and records new timestamp. Grants repeatQB on both reaccept (+1) and completion (+1) for +2 per full cycle.
- **Questgiver blacklist via EmoteManager prefix** — To blacklist specific NPCs from granting repeat rewards, patch `EmoteManager.ExecuteEmote` prefix for `StampQuest`/`StampMyQuest` to capture the giver WCID in a `[ThreadStatic]` variable, then read it in the `QuestManager.Update` postfix. Store on player as `PropertyInt` for persistence across the tick.
- **Daily fixed-time event scheduling** — For daily resets at a specific local time (e.g. 9pm CST), use `TimeZoneInfo.FindSystemTimeZoneById("Central Standard Time")` to convert local target time to UTC, then compare `DateTime.UtcNow` against that. Full clear + repopulate on rotation; autofill from next tier down if insufficient eligible quests exist.

- **Enlightenment pool expansion formula** — `GetEnlightenmentMultiplierFactor` returns `1 + pool + augmentBonus + achievementBonus + equipmentBonus`. `pool` = `FakeFloat 11012` (base enlight + challenge quit contributions + achievement unlocks). `augmentBonus` = `player.AugmentationBonusXp * 0.05f` (Quick Learner). `equipmentBonus` = `player.EnchantmentManager.GetXPBonus()` (Augmented Understanding trinket enchantments, 3 levels × 2%).
- **ACE vanilla `SpellLevelProgression` blast spells contain `SpellId.Undef` at indices 0–1** — AcidBlast, ShockBlast, FrostBlast, LightningBlast, and others have `Undef` for spell levels 1–2 because those spells don't exist in retail. When `SpellLevelChance.Roll(tier)` returns 1 or 2 for Tier 1 loot, `RollSpellLevels` adds `SpellId.Undef` (0) to the item spellbook, causing `Spell.Init(0)` DEBUG spam during `GetMaxBaseMana` / `GetMaxSpellPower`. Always filter `Undef` from rolled spell lists before adding to spellbooks. `EmpyreanAlteration/Features/MutatorHooks.cs:PostRollSpellLevels` provides a Harmony postfix fix.
- **Donated-offset pattern for tracked currency spending** — When a player "spends" a tracked currency that must remain visible in the system (e.g., quest registry base QP), use a `Donated*Offset` property. Subtract the offset at display time. This preserves the original tracked data while making the spend visible and permanent. Example: `DonatedQuestPoints` offset reduces effective QB without removing actual quest solves.
- **Lottery prize grants must bypass contribution tracking** — When a prize system re-grants a currency that also feeds itself (e.g., QB prizes funded by QB donations), the grant method must bypass `AddExtraQuestPoints` and write directly to the underlying property. Otherwise, each prize creates a recursive 10% contribution back into the pool. Example: `GrantLotteryQbPrize` directly sets `QuestPointsExtra`; `AddExtraQuestPoints` only tracks positive earned amounts for contributions.
- **Dual-pool prize design for mixed-value currencies** — When one currency (pyreals) and another (QB) both contribute to a prize pool but have different transfer semantics, maintain separate pools. Pyreal pool is additive from multiple sources; QB pool is direct 1:1 from donations. Split both pools by the same winner fractions. This avoids conversion-rate ambiguity and preserves the feeling of "winning actual QB."

- **ACE `ItemLevel` is a read-only computed property** — It derives from `ItemTotalXp`, `ItemBaseXp`, `ItemMaxLevel`, and `ItemXpStyle` via `ExperienceSystem.ItemTotalXPToLevel`. Do not assign to `item.ItemLevel`; instead compute the target XP with `ExperienceSystem.ItemLevelToTotalXP` and set `item.ItemTotalXp`. EmpyreanAlteration patches this method for custom curves; wrap calls in `ItemXpCurveContext.Enter(item)` / `.Leave()` so the prefix attributes the math to the correct item.
- **Cross-mod Harmony patches without assembly references** — When patching a type from another mod (e.g., `Overtinked.ChaosFailureEffects`), use `AccessTools.TypeByName("Namespace.Type")` inside a static `TargetMethod()` method, with `Prepare()` returning `false` if the type is absent. This avoids a hard project reference and gracefully skips patching if the target mod is unloaded. Remember to manually `PatchCategory` / `UnpatchCategory` in the host mod's `RefreshAll()` or equivalent lifecycle method.
- **Initialize `Settings` before `base.Start()` in `BasicPatch<>` subclasses** — `base.Start()` applies Harmony patches. If a patch signature is wrong and throws, `Start()` aborts before `Settings` is assigned, leaving postfixes active with a null settings reference. Assign `Settings = SettingsContainer.Settings ?? new Settings()` before calling `base.Start()`.
- **`PropertiesEmoteAction.Type` is a `uint`, not `EmoteType`** — The ACE `PropertiesEmoteAction` model stores the emote type as `uint Type { get; set; }`. Always cast with `(EmoteType)emote.Type` rather than accessing `emote.EmoteType`.
- **Static `HashSet<T>` is not thread-safe — use `ConcurrentDictionary<T, byte>` instead** — `HashSet.Add` / `.Remove` can throw `IndexOutOfRangeException` when called concurrently from multiple threads (e.g., `WorldManager` tick handlers). Replace with `ConcurrentDictionary<T, byte>` and use `TryAdd(key, 0)`, `TryRemove(key, out _)`, and `ContainsKey(key)`. This matches the existing pattern in `SummoningClasses.cs` (`TrackedPetGuids`).

ItemType.None) & target.ItemType) == ItemType.None` and blocks the interaction with `SendTransientError`. - **AC client uses Windows-1252 encoding — avoid Unicode em-dashes and smart quotes in weenie strings.** — The AC client renders `weenie_properties_string` values using Windows-1252. Unicode characters like `—` (em-dash, U+2014) and `“”` (smart quotes) display as garbled symbols. Always use ASCII equivalents (`-`, `""`) in SQL that populates `weenie_properties_string`.
- **Bidirectional use-on-target UX pattern.** — When implementing item-on-item interactions, support both directions (`Tool → Target` and `Target → Tool`) to match player intuition. - **Biota cleanup required for existing items** — ACE copies weenie properties to each item's biota in `ace_shard` at creation time. Updating the world template in `ace_world` only affects **future drops**; items already in player inventories retain the old properties. After applying world template SQL, always run a matching biota cleanup against `ace_shard` so existing items are patched in-place. No restart needed for biota changes.
- **MySQL `UPDATE` silently fails when row doesn't exist** — `weenie_properties_int` has a `UNIQUE KEY` on `(object_Id, type)`. Using `UPDATE ... WHERE object_Id = X AND type = Y` when no row exists silently does nothing. Use `INSERT INTO ... ON DUPLICATE KEY UPDATE` for idempotent SQL that works whether the row is present or absent. Example: `INSERT INTO weenie_properties_int (object_Id, type, value) VALUES (42516, 94, 128) ON DUPLICATE KEY UPDATE value = 128;`
- **Verify DB state after applying SQL** — Always query the database to confirm changes took effect, especially when troubleshooting "nothing changed" reports. The server may be running old cached weenies even after SQL is applied. `SELECT type, value FROM weenie_properties_int WHERE object_Id = 850200 AND type = 94;`

- **MaterialType-based dynamic renaming** — Read `item.MaterialType`, call `.ToString()` for the friendly material name, then replace that exact word in `item.Name`. Store the original name in a custom `PropertyString` (e.g., `(PropertyString)11033`). Fallback to prepending the prefix if the material name is not found in the item name.
- **Coalesced Mana tiered progression IDs** — Lesser (WCID 42516, max level 25), Greater (WCID 42517, max level 50), Aetheric (WCID 42518, max level 75). Banked via LeyLineLedger properties 41100-41102. Use `SkillCheck.GetSkillChance((int)arcaneLore.Current, difficulty)` + `ThreadSafeRandom.Next(0.0f, 1.0f)` for cosmetic skill checks that clear the cursor.
- **`SettingsContainer.Settings` may fail to deserialize `List<string>` properties** — ACE.Shared's `SettingsContainer` can return empty lists even when `Settings.json` contains valid array data. Always verify deserialized collections before using them. If empty, manually deserialize with `JsonSerializer.Deserialize<Settings>(File.ReadAllText(path))` and assign back to `SettingsContainer.Settings`. Example: `ValheelContent/PatchClass.cs` manual deserialize workaround for `DisabledContentFolders` / `EnabledContentFiles`.
- **`CreatureVital.MaxValue = Ranks + StartingValue + attr_bonus`** — Setting `creature.Health.Ranks = desiredMax` does NOT make the creature full health. `MaxValue` includes the base `StartingValue` and attribute formula bonus on top of `Ranks`. After scaling `Ranks`, always call `creature.SetMaxVitals()` to fill current vitals to the true computed maximum. Example: `Swarmed/PatchClass.CallForHelp.cs` and `CreatureVariation.cs`.
- **`RadarColor.Cyan` (0x09) is the enum value for teal/admin dots** — The `RadarColor` enum defines `Cyan = 0x09` and aliases `Admin = Cyan` / `Sentinel = Cyan`. The `WorldObject.RadarColor` property is `RadarColor?` (nullable). Assign directly: `RadarColor = RadarColor.Cyan;`. No broadcast needed at spawn time if set before object creation packet is sent.
- **Quest trophy drops need explicit creature-type validation** — A flat `List<uint> QuestItemWcids` rolled randomly on every kill will drop tusker tusks from drudges and olthoi pincers from ursuins. Always validate the dying creature's `WeenieClassId` against a per-trophy whitelist before creating the item. Example: `BetterSupportSkills/Skills/TrophyDrops.cs:IsValidTrophySource()`.
- **`/god` in ACE sets `IsAdmin`/`IsInvulnerable` but does NOT change `player.Level`** — Mobs scaling via `player.Level ?? 1` still read the real level (e.g., 275 for a level-capped admin). If you want scaling to match the perceived god level, add an `AdminLevelOverrideProperty` (custom `PropertyInt`) and read it with priority when `player.IsAdmin` is true.
- **`ServerMaxLevel` must read `DatManager.PortalDat.XpTable.CharacterLevelXPList.Count` at call time** — Caching it in a static field at class initialization captures the vanilla 275-level table before AureatePath's `OnWorldOpen` extension runs. Always read the count inside the scaling method so it reflects runtime XP table length.
- **ACE `UiEffects` = world shader + inventory glow — not just 3D** — `UiEffects = 256` (`Acid`) gives a green glow identical to retail acid weapons. `Magical (1) | Acid (256) = 257` combines blue + green. Other useful values: `Fire=32` (red), `Lightning=64` (purple), `Frost=128` (white). **Important: `BoostHealth | BoostStamina = 20` (bitwise OR of 4 + 16) renders as a red outline in the AC client despite the individual flag names suggesting yellow.** Set via `PropertyInt.UiEffects` and broadcast `GameMessagePublicUpdatePropertyInt`.
- **Item property changes need `CalculateObjDesc()` + `GameMessageUpdateObject` for client refresh** — Setting `Name`, `UiEffects`, `ItemMaxLevel`, etc. on a `WorldObject` updates server state but the client won't refresh its examine panel or 3D appearance until you call `item.CalculateObjDesc()` followed by `EnqueueBroadcast(new GameMessageUpdateObject(item))`.
- **Patch property getters for client + server consistency on shared data** — If a `PropertyFloat`/`PropertyInt` affects both client display (via `GameEventApproachVendor` or other packets) and server math, patch the property getter (e.g., `Vendor.BuyPrice`) with `[HarmonyPatch(..., MethodType.Getter)]` instead of only patching the downstream calculation method. This ensures client UI and server behavior stay in sync without duplicating logic. Example: LeyLineLedger `VendorSellRate` patches `Vendor.get_BuyPrice` so the sell-price packet and `GetBuyCost` both see the same reduced rate.
- **ACE server port collision: each server binds `Port` and `Port+1`** — ACE `SocketManager` creates two listeners per server: `Port` (primary) and `Port+1` (secondary/browser). Running a test server on 9000 and a live server on 9001 causes the live server's primary (9001) to collide with the test server's secondary (9001). Always separate servers by at least 2 ports: e.g., test=9000, live=9002 (or 9010).
- **Verify deployed DLL timestamps before restarting a production server** — After `dotnet build`, output goes to `C:\ACE\Mods\`. If you run two servers (test + live), copying DLLs to the wrong directory or forgetting to copy causes the old build to run on restart. Always `ls -la` the deployed DLL and confirm the mtime matches the repo build. Example: EmpyreanAlteration NRE fix was in repo but old DLL (Apr 29 01:54) was on live; server crashed on startup until fresh DLL copied.
- **Town Network is dungeon landblock 0x0007, not surface 0xA9B4** — The Town Network interior is a dungeon at landblock `0x0007` with cells like `0x0007010B`. Landblock `0xA9B4` is the surface cell where the TN portal drops players. When placing objects *inside* the TN, use `0x0007` GUIDs and cells.
- **`INSERT INTO ... ON DUPLICATE KEY UPDATE` for idempotent weenie property changes** — `weenie_properties_int` has a `UNIQUE KEY` on `(object_Id, type)`. `UPDATE ... WHERE` silently does nothing if the row doesn't exist. Use `INSERT INTO weenie_properties_int (object_Id, type, value) VALUES (X, Y, Z) ON DUPLICATE KEY UPDATE value = Z;` for SQL that works whether the row is present or absent.

  - Host: 127.0.0.1, Port: 3306

- **Vendor inventory injection — use WCID detection, NOT inventory scanning** — Detecting vendor type by scanning `vendor.DefaultItemsForSale` fails when the inventory is empty at approach time (before rotation). Always detect by vendor `WeenieClassId` or `MerchandiseItemTypes` bitmask directly. Example: `AcademyBlacksmithWcids.Contains(vendor.WeenieClassId)` or `(merchTypes.Value & (int)ItemType.Armor) != 0`.
- **Harmony prefix priority for vendor approach ordering** — When multiple mods inject items into vendor inventory on `Vendor.ApproachVendor`, set priorities so the mod that CLEARS inventory runs first (`Priority.First`) and the mod that ADDS static items runs last (`Priority.Last`). Example: QOL VendorLootRotation clears+generates at `Priority.First`; EmpyreanAlteration (or any static injector) adds items at `Priority.Last`.
- **Cross-mod vendor pricing via reflection bridge** — When one mod (e.g., EmpyreanAlteration) needs another mod's (LeyLineLedger) economy multiplier without a project reference, use `AppDomain.CurrentDomain.GetAssemblies()` + `GetType()` + `GetMethod()` + `Invoke()`. Always wrap in try/catch with fallback (default 1.0x). Cache the reflected MethodInfo if called frequently.
- **UiEffects values for item visual overlay** — ACE `UiEffects` enum (PropertyInt=18) controls both 3D shader and inventory icon glow:
  - `Magical=1` (blue), `Poisoned=2` (green), `BoostHealth=4` (yellow), `BoostMana=8` (blue), `BoostStamina=16` (yellow)
  - `Fire=32` (red), `Lightning=64` (purple), `Frost=128` (white), `Acid=256` (green), `Nether=4096` (dark purple)
  - Combine with bitwise OR: `BoostHealth | BoostStamina = 20` (yellow/gold). Set on item, then broadcast `GameMessagePublicUpdatePropertyInt(item, PropertyInt.UiEffects, value)`.
- **Single source of truth for vendor pricing** — Only ONE mod should control vendor buy prices. When moving pricing from QOL to LLL, disable the old feature (`EnableVendorPriceInflation: false`) in QOL Settings.json and add the new pricing system to the owning mod (LLL). Prevents double-pricing bugs.
- **Inventory injection tracking — check inventory directly, not static state** — Static `HashSet` tracking of injected items fails when another mod clears the vendor inventory. Instead, check `vendor.DefaultItemsForSale.Values.Any(item => item.WeenieClassId == wcid)` each approach to re-inject items that were cleared.
- **`Assembly.Location` can be empty string when the server loads from a different working directory.** — `typeof(MyClass).Assembly.Location` returns `""` if the DLL was loaded from a path relative to the server's cwd. Use `ModManager.ModPath` as the canonical mod directory for sidecar files (JSON, SQL, etc.). Example: `var modDir = Path.GetDirectoryName(typeof(PatchClass).Assembly.Location); if (string.IsNullOrEmpty(modDir)) modDir = Path.Combine(ModManager.ModPath, "ModName");`
- **`ItemType.Shield` does NOT exist in the ACE enum** — Shields are classified as `ItemType.Armor`. Attempting to use `ItemType.Shield` causes `CS0117` at compile time. Always use `ItemType.Armor` for shields in equipment checks.
- **Gracefully deprecating a `Features` enum value** — When renaming a feature (e.g., `GiveNpcSingleFromStack` → `NpcStackTurnIn`): (1) Add the new enum value, (2) Keep the OLD enum value in the `Features` enum definition for JSON backward compatibility, (3) Add a DEPRECATED doc comment to the old toggle in `Settings.cs`, (4) In `PatchClass.RegisterEnabledPatchCategories()`, check BOTH toggles: `if (Settings.EnableNewFeature || Settings.EnableOldFeature) enabledFeatures.Add(Features.NewFeature);`. After a migration period, the old enum can be safely removed.
- **Reflection over `IEnumerable` / `IList` uses non-generic interfaces for unknown types** — When iterating over `Settings.Items` from another mod via reflection, `itemsProp.GetValue(settings) as IEnumerable` (not `IEnumerable<T>`) and `variantProp?.GetValue(item) as IList` (not `IList<T>`) are the correct casts because the generic type arguments are not known at compile time. Casting to `IEnumerable<T>` causes `CS0305`.
- **Never assume AC enum or ID values are sequential** — ACE `SpellId`, `PropertyInt`, `WCID`, and other ID spaces have arbitrary gaps, manual assignments, and historical offsets. Doing `(uint)(SpellId.HealSelf1 + 3)` to guess `HealSelf4` will silently produce the wrong spell or crash `Spell.Init()`. Always use hardcoded arrays of known-valid IDs, query the database, or reference the actual enum definition. This applies to spells, properties, emote types, and any dat-file-derived ID.
- **Always check `.references/` ACE source before guessing** — When investigating runtime behavior (e.g., "why does Value=500 when DB says 100"), grep the ACE source directly rather than speculating. Key files: `WorldObject_Properties.cs`, `WorldObjectFactory.cs`, `WeenieConverter.cs`, `Vendor.cs`, `Player_Commerce.cs`. The source is the single source of truth for property initialization, caching, and calculations. Guessing leads to wasted time and wrong fixes.
- **LeyLineLedger is the single source of truth for all bank, vendor, and pricing systems** — No other mod may implement vendor buy/sell price multipliers, banked currency handling, or vendor approach packet rewriting. All pricing logic lives in LLL. Other mods that need pricing adjustments must expose a public API and be called by LLL via reflection bridge, never by patching the same Harmony targets.
- **Foolproof imbue salvage items must NOT stack.** Setting `MaxStackSize` on foolproof items (e.g., Foolproof Aquamarine, Foolproof Black Garnet) causes them to break or behave incorrectly. In `LootRoller.CreateItemFromCategory`, explicitly exclude items whose `Name.Contains("Foolproof")` from `MaxStackSize` assignment. Same applies to any other non-stackable modded items.
- **Runtime JSON configs override compiled defaults.** `DefaultLootConfig.cs` is a fallback only. The active config is `Loremaster/LootConfig.json` (or whatever `LootConfigPaths.ResolveLootConfigPath` returns). Any changes to `DefaultLootConfig.cs` must be mirrored in the JSON or they won't appear at runtime.
- **BetterLootControl is the single source of truth for all loot tables.** Any mod needing bonus loot pools (Loremaster repeat-solves, Lockboxes, WorldEvents placement rewards) should reference `BetterLootControl` (the consolidated mod absorbing `SharedLoot` + `BetterChestLoot`), not individual mods.
- **Use `PropertyInt` fallback when dynamically assigning ratings by string name.** — `WorldObject` exposes some ratings as nullable `int?` properties (e.g., `DamageRating`, `CritDamageRating`) while others are only accessible via `GetProperty(PropertyInt)`. When applying ratings dynamically by name (e.g., from JSON config), try reflection on the property first, then fall back to a `switch` mapping string→`PropertyInt`. This handles both patterns without compile-time dependency on every possible rating.
- **Generic rating scaling across ALL item types** — When item ratings need to level up regardless of category (weapon/armor/jewelry/shield), add a `TryScaleExistingRatings` step after the category-specific `ApplyLevelUp` effects. Iterate over known `PropertyInt` ratings, check if current > 0, increment by configurable amount every N levels. This ensures ratings generated by any mod (BLC, EA, etc.) scale consistently.
- **Harmony patching `override` methods in partial classes fails** — `Creature.Heartbeat` is an `override` defined in `Creature_Tick.cs` (partial class). Patching it directly causes "Patching exception in method virtual System.Void ACE.Server.WorldObjects.Creature::Heartbeat". The base `virtual` is `WorldObject.Heartbeat` in `WorldObject_Tick.cs`. Always patch the base `virtual` method; use `WorldObject __instance` parameter and `is` cast to filter in the postfix. A single bad patch target can prevent the ENTIRE mod from loading.
- **MySQL `weenie_properties_int` type numbers are not self-documenting** — `type = 11` = `MaxStackSize`, `type = 16` = `ItemUseable`. Using `16` for stack size silently does nothing (clients still see stack size 1). Always verify with `ACE.Entity.Enum.Properties.PropertyInt` enum before writing SQL that manipulates weenie properties.
- **Consolidated deposit feedback pattern** — When a command like `/bank deposit all` runs multiple subsystems (cash, luminance, items, salvage, denominated families), each subsystem should return `bool` (success/any deposited). The command handler aggregates all bools and sends exactly ONE message: either the list of what was deposited, or `"You have nothing eligible for deposit."`. This prevents the 5-message spam that trained users to ignore deposit output.
- **Client examine panels read from `AppraiseInfo` packets, not live properties** — A Harmony postfix on `WorldObject.Attuned` (getter) makes items droppable on the server, but the client still displays "Attuned" in the examine panel because it reads from the `weenie_properties_int` snapshot sent in the appraisal packet. To hide the attuned status visually, also patch `AppraiseInfo.BuildProperties` (or the specific property after `BuildProperties` runs) to override `PropertiesInt[PropertyInt.Attuned]` before the packet is serialized.
